# seed.js 重構紀錄 — 台灣50清單修正 + 假資料全面換成 TWSE 真實K線

**日期：** 2026-06-01  
**分支：** `backend-db`  
**異動檔案：** `backend/seed.js`（唯一異動，不影響其他檔案）

---

## 一、改之前有什麼問題

### 問題一：台灣50清單是錯的

**初版錯誤：** seed.js 直接複製 `frontend/src/data/twStocks.js` 前 50 筆。那份前端清單是教學練習用，不是真正的 0050 成分股。  
結果：2474 可成混進去、2345 智邦不見了。

**修正後仍有誤：** 更換後的清單標注來源為 TAIFEX 衍生商品頁面，並非 0050 成分股的正確來源。  
用元大投信官方 CSV 比對後，仍有 7 支錯誤、漏了 7 支。

### 問題二：K 線全部是假的

舊版用 Mulberry32 偽亂數 + GBM 數學公式模擬 300 根假 K 線。數字和真實市場無關。  
首頁一開始顯示的全是虛構股價，要等使用者點進個股才會換成真實資料。

### 問題三：stock_sync 沒被寫入

`dao.js` 靠 `stock_sync` 判斷「這支有沒有真實資料」。舊版 seed 不寫這張表，導致 dao.js 每支都認為需要重抓，加重載入壓力。

### 改動二：假資料全部移除，改抓 TWSE 真實K線

移除的假資料程式碼：mulberry32()、boxMuller()、generateDailyBars()、buildTradeDates()、BARS_PER_STOCK、SEED_END_DATE、TOP50 陣列中的 price 和 vol 欄位。

改為呼叫 `fetchRecentHistoryWithFallback(code, 2)` 向 TWSE 抓最近 2 個月真實日 K 線。

**為什麼 2 個月：** 50 支 × 13 個月 = 650 次請求需 3–5 分鐘，2 個月只需 40–60 秒，夠顯示 MA20。使用者點進個股後 dao.js 自動補 13 個月。

**現在的資料時間線：**
```
seed 跑完 → 50 支各有約 21 根真實 K 線 + stock_sync.source='twse'
使用者看首頁 → dao.js 直接讀 DB，顯示真實 K 線（不再顯示假的）
使用者點進個股 → dao.js 在背景補完整 13 個月歷史
```

### 改動三：新增 writeBarsAndSync() 

每支股票抓到真實 K 線後，統一做三件事（包在同一個 transaction）：
1. DELETE 該股舊 K 線（清掉可能殘留的假資料）
2. INSERT 新的真實 K 線
3. UPSERT stock_sync（標記 source='twse'，timestamp=現在）

### 改動四：新增 validateTop50() 寫入前驗證

每次 seed 執行前自動驗 5 件事，任一失敗 → 程式停止，DB 完全不動：
1. 陣列長度剛好 50
2. 無重複代號
3. 無空白代號或名稱
4. 2345 智邦必須存在
5. 2474 可成不可存在

### 改動五：修正 --force 資料遺失 Bug

```
修正前：DELETE 三張表 → validateTop50()  ← 驗證失敗 = DB 被清空但沒重寫
修正後：validateTop50() → DELETE 三張表  ← 驗證失敗 = DB 完全不動
```

### 改動六：--sync-stocks 加入自動下架邏輯

以前：移除成分股後跑 --sync-stocks，舊股票仍維持 is_active=1（仍出現在搜尋）  
現在：--sync-stocks 結尾自動把「不在 TOP50 清單中的 active 股票」設成 is_active=0

**下次成分股調整只需要：**
1. 修改 TOP50 陣列
2. 跑 `node seed.js --sync-stocks`（自動補新股、下架舊股、抓 K 線）

### 改動七：修正說明文件中的錯誤 URL

原本：`https://www.yuantaetfs.com/#/Product/Detail/0050`（hash route，不穩定）  
修正：`https://www.yuantaetfs.com/product/detail/0050/ratio`（直接持股比重頁）

## 四、已知限制（設計取捨，不是 Bug）

| 限制 | 說明 |
|---|---|
| seed 後每支只有約 21 根 K 線 | 只抓最近 2 個月，當月尚在進行中所以只有上個月完整資料。MA60/MA240 需等使用者點進股票後 dao.js 在背景補足 |
| stocks.base_price / volatility = 0 | 假資料時代留下的欄位，新版不用。`searchDbStocks` 回傳的 `price` 欄位為 0，不影響即時股價（由 getQuote 抓取） |
| TWSE 離線時 seed 的股票沒有 K 線 | 無 fallback。失敗清單會印在終端機，之後跑 `--sync-stocks` 補抓 |

## 五、給其他組員的 AI：驗收這次改動

把以下內容貼給你的 AI：

---

請幫我在 SSMS 執行以下 SQL，並告訴我每一段的結果是否符合預期：

```sql
-- 驗收1：active 股票總數應該剛好 50
SELECT COUNT(*) AS active_stocks FROM dbo.stocks WHERE is_active = 1;
-- 預期：50

-- 驗收2：2345 智邦存在且 active
SELECT code, name, is_active FROM dbo.stocks WHERE code = '2345';
-- 預期：找到一筆，is_active = 1

-- 驗收3：2474 可成不是 active
SELECT code, name, is_active FROM dbo.stocks WHERE code = '2474';
-- 預期：找到一筆，is_active = 0

-- 驗收4：K 線全部來自 TWSE
SELECT source, COUNT(*) AS stock_count FROM dbo.stock_sync GROUP BY source;
-- 預期：source = 'twse'（count 可能超過 50，因為下架的舊股票也有記錄）

-- 驗收5：K 線是真實近期交易日，不是假資料
SELECT TOP 5
    s.code,
    CONVERT(varchar, b.trade_date, 23) AS trade_date,
    b.[close]
FROM dbo.stock_daily_bars b
JOIN dbo.stocks s ON s.id = b.stock_id
WHERE s.code = '2330'
ORDER BY b.trade_date DESC;
-- 預期：trade_date 是最近的真實交易日（2026-05 前後），close 接近台積電真實股價

-- 驗收6：下架的舊錯誤股票確認不在 active 清單
SELECT code, name FROM dbo.stocks
WHERE code IN ('3443','4958','8046','3481','3665','3189','6770')
  AND is_active = 1;
-- 預期：0 筆
```

如果以上全部符合預期，代表 seed.js 的修正已正確套用到資料庫。

```bash
cd backend

```

這個指令安全（不刪 orders/holdings），會自動：補新成分股、下架舊成分股、抓 TWSE K 線。
