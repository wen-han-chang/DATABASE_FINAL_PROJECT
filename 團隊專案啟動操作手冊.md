## 本次更新重點（給團隊快速看）

這次更新主要讓「個股查詢圖表」更接近正式系統，而不是依賴展示用測試資料。

| 更新項目 | 現在怎麼運作 | 團隊要知道什麼 |
|----------|--------------|----------------|
| K 線圖資料 | 後端優先回 SQL Server 已存的 TWSE 歷史資料 | 如果資料庫已經有舊資料，點股票會直接顯示，不用等完整重抓 |
| 首次查新股票 | 如果資料庫還沒有 TWSE 歷史，後端先抓最近約 2 個月真實資料 | 使用者能先看到近期真實 K 線，不必等 13 個月資料都抓完 |
| 背景補資料 | 前端顯示後，後端再補最近 2 個月或完整 13 個月 | 補完後圖表會更新；使用者不需要手動操作 |
| 載入提示 | 只有完全沒有 K 線資料時才顯示「真實歷史資料載入中」 | 只要圖表已有資料，就不應該一直顯示載入中 |
| 畫面字樣 | K 線與下單頁的顯示文字不再使用舊版測試資料字樣 | 避免產品展示時讓使用者誤以為 K 線是測試行情 |
| 搜尋 bug | `2382 廣達` 這種輸入可以正確找到股票 | 已選到股票時，不會再出現「找不到」的下拉訊息 |

摘要：舊 TWSE 歷史已經在資料庫時直接用；沒有舊資料才先抓短期真實資料；完整資料再背景補。這樣能縮短等待時間，也能清楚標示資料來源。

## 先看這段：股票資料到底從哪裡來

本專案不是完全沒用 `https://openapi.twse.com.tw/`。  
正確說法是：**有用，但不是所有股票資料都靠它。**

原因很簡單：台灣證交所不同網站提供的資料用途不同。後端把它們分工使用，不硬把同一個 API 拿來做所有事情。

| 資料需求 | 後端使用的來源 | 為什麼這樣用 |
|----------|----------------|--------------|
| 股票清單、最新交易日資料、首頁 TWSE 驗證 | `https://openapi.twse.com.tw/` | 這個 OpenAPI 適合拿最新交易日與市場資料，但不適合一次拿每檔股票長期歷史 K 線 |
| 歷史 K 線、MA 指標需要的日線資料 | `https://www.twse.com.tw/exchangeReport/STOCK_DAY` | 這個來源可以查單一股票、單一月份的歷史日線；後端有舊資料就直接用，沒有才先抓短期資料再背景補完整資料 |
| 圖表上方的準即時現價、漲跌、開高低量 | `https://mis.twse.com.tw/stock/api/getStockInfo.jsp` | 這個來源比較適合查單檔準即時報價，後端只在使用者看某檔或按更新時才抓，並做 20 秒快取 |

流程摘要如下：

```txt
前端想畫 K 線
  -> 向後端要某檔股票的歷史資料
  -> 如果資料庫已有 TWSE 歷史資料，直接顯示完整舊資料
  -> 如果資料庫還沒有 TWSE 歷史資料，才先抓近期真實日線讓畫面先有結果
  -> 前端再發一個背景請求補資料
  -> 已有舊資料時只補最近約 2 個月；完全沒有歷史時才補最近約 13 個月
  -> 補完後存進 stock_daily_bars，前端自動重畫 K 線與 MA

前端想顯示目前股價
  -> 向後端要某檔股票的準即時報價
  -> 後端先看 20 秒快取
  -> 沒快取才去 TWSE MIS 抓
  -> 前端只把它顯示在 K 線圖上方，不把它硬塞進歷史 K 線
```

這樣取捨的原因：

- 歷史 K 線是「正式交易日資料」，適合存進資料庫並拿來算 MA。
- 準即時報價是「現在看到的價格」，只適合顯示目前狀態，不適合當成正式日 K 收盤資料。
- 使用者沒看的股票不需要馬上抓，否則會浪費官方與本機資源。
- 如果資料庫已有歷史資料，畫面會直接顯示，不應該一直顯示載入中；只有完全沒有真實 K 線時才會先抓近期資料。
- `openapi.twse.com.tw` 仍然有用，只是它不是歷史 K 線與準即時報價的主要來源。

## 先看這段：資料表有沒有違反前端資料表參考文件

沒有違反。前端資料表參考文件 `專題資料庫_ai建議.txt` 是 MySQL 寫法，本專案實際使用 SQL Server，所以 `database/schema.sql` 把語法轉成 SQL Server 可執行版本。

資料表用途沒有被亂改：

- 使用者、問卷、股票、K 線、投資組合、持股、交易紀錄、AI 決策卡、知識庫這些核心表都有保留。
- `AUTO_INCREMENT`、`ENUM`、`BOOLEAN`、`ON UPDATE CURRENT_TIMESTAMP` 這些 MySQL 語法，在 SQL Server 不能直接用，所以改成 `IDENTITY`、`CHECK`、`BIT`、trigger。
- 後端額外新增 `stock_sync`，它不是前端原本設計的業務表，而是後端用來記錄「某檔股票今天有沒有同步過 TWSE 歷史資料」的維護表。
- `stock_sync` 不會破壞前端設計，也不會改變原本 13 張核心資料表的用途。

## 0. ★ 先看：為什麼要這樣做、每次都要嗎、是不是真資料

### 0-1 啟動時打的每個指令是什麼、每次展示或測試都要嗎

| 指令 | 作用 | 每次展示或測試都要？ |
|------|------|------------------|
| `cd /d 路徑` | 切換到該資料夾（進去才能跑後面指令） | 要 |
| `npm install` | 安裝程式需要的零件（套件） | **不用**，只第一次、或換電腦、或別人第一次 clone 後 |
| `node seed.js` | 把 50 檔股票 + 15000 筆 K 線 + 展示帳號灌進資料庫 | **不用**，只第一次（已灌過會自動略過） |
| `npm run dev` | **啟動伺服器**（後端 / 前端各要跑一次） | **每次都要** |

### 0-2 每次展示或測試的最短流程（熟了之後看這個就好）

```
視窗 A：從專案根目錄進入 `backend` → `npm run dev`
視窗 B：從專案根目錄進入 `frontend` → `npm run dev`
瀏覽器：http://localhost:5173 → 登入 demo@invest.ai / demo123
展示或測試結束：兩個視窗各按 Ctrl + C
```

> ⚠️ **後端、前端兩個伺服器必須「同時都開著」**，關掉任何一個，網頁就抓不到資料。
> 這就是為什麼要開「兩個視窗」。

### 0-3 為什麼要這樣做（流程說明）

- 「後端」是去 TWSE 抓資料、連資料庫處理註冊買賣的程式 → 視窗 A，網址 `localhost:3001`。
- 「前端」是你看到的網頁畫面 → 視窗 B 啟動，瀏覽器開 `localhost:5173`。
- 網頁（前端）自己沒有資料，要跟後端要；後端沒開，網頁就是空的或顯示「失敗」。

### 0-4 是不是真資料（對展示對象 / 團隊一定要講對）

| 你在畫面看到的 | 真 / 假 | 說明 |
|----------------|---------|------|
| 首頁「TWSE 市場資料」區塊 | **真實** | 後端當下去 TWSE 官方 OpenAPI 抓的，但官方只給「最新一個交易日」，不是盤中即時跳動、也沒有真實多年歷史（官方限制，非故障） |
| K 線圖（歷史日線、MA 指標） | **真實 TWSE 歷史** | 若資料庫已有 TWSE 歷史，直接從 SQL Server `stock_daily_bars` 顯示；若資料庫沒有，先抓最近約 2 個月真實日線，再背景補最近約 13 個月。前端不再用測試 K 線當圖表備援 |
| K 線圖上方「即時報價條」 | **真實準即時** | 後端打 TWSE MIS 報價（延遲數秒～20 秒），顯示現價/漲跌/開高低量。使用者進入或換股票才抓一次，可按「🔄更新」，後端 20 秒快取 |
| 買賣定價 | 用上述歷史/報價 | 下單寫進 orders/holdings，仍是真實資料庫交易 |
| 註冊 / 登入 / 買賣 / 持股 / 交易紀錄 | **真實資料庫操作** | 真的寫進 SQL Server `ncu_db`，資料庫課程評分重點在這，不在行情真不真 |

> 標準說法：「行情看板、歷史 K 線、準即時報價都來自 TWSE 官方來源；
> 歷史 K 線在使用者查看某檔時才抓取並存入 SQL Server，準即時報價在使用者查看或手動更新該檔時才抓取，後端有 20 秒快取；
> 帳號、問卷、買賣、持股與交易紀錄都真的寫進 SQL Server 資料庫。種子資料保留為資料庫初始化與測試用途，不再被前端 K 線圖當成真實行情展示。」

## 2. 重要觀念：資料哪些是真的、哪些是測試資料（一定要懂，會被問）

現在系統的資料分三種，請務必分清楚：

| 資料 | 真實／測試 | 來源 | 哪裡用到 |
|------|---------|------|----------|
| K 線歷史日線 | **真實** | 看某檔時，後端去 TWSE 官方 `STOCK_DAY` 抓最近約 13 個月真實日線，存進資料庫 `stock_daily_bars` | 首頁／查詢頁 K 線圖、MA 均線 |
| 即時報價 | **真實（準即時）** | 後端打 TWSE MIS 報價（延遲數秒～20 秒；收盤後＝最新收盤） | K 線圖上方的報價條 |
| 帳號／問卷／買賣／持股 | **真實資料庫操作** | 真的寫進 SQL Server `users/investor_profiles/portfolios/holdings/orders` | 登入、問卷、下單 |
| 股票「起始參考價、波動係數」、seed 產生的初始 K 線 | **測試資料** | `seed.js` 灌入的種子（見 2-1） | 用於資料庫初始化、下單練習參考價與離線測試；前端 K 線圖不再把它當成真實行情展示 |

重點：**真正的歷史與報價是 TWSE 官方真資料；seed 測試資料只剩「初始化、測試、下單練習參考」的角色**
（下一段詳細解釋 seed）。

### 2-1 「種子資料 seed」是不是真的？做什麼用的？（先前沒講清楚，補上）

`node seed.js`（或啟動器的 `install-first.bat`）會灌三樣東西進資料庫：

| seed 灌的東西 | 是不是真的？ | 作用（為什麼需要它） |
|---------------|--------------|----------------------|
| 25 個產業別 + 50 檔股票（代碼、名稱、產業） | **代碼/名稱是真的台股**（台積電 2330 等），但**起始價、波動係數是測試參數** | 定義「系統有哪些股票可選、屬於哪個產業」。這是 `stocks`/`sectors` 兩張表的基礎，沒有它，買賣的外鍵、產業 JOIN、選股都無法運作 |
| 每檔 300 筆日 K（共 15000 筆） | **測試資料**（用 GBM 亂數演算法產生，程式有標示） | 現在主要是資料庫初始化與離線測試用途；前端 K 線圖已改成只顯示 TWSE 真實資料或明確載入狀態 |
| 展示帳號 `demo@invest.ai`／`demo123` | 帳號列是**真的資料庫資料**，但它是給產品展示與測試用的帳號 | 讓測試人員不用重新註冊就能登入查看展示資料 |

**關鍵觀念（一定要懂）**：
- seed 的 15000 筆 K 線**不是真股價**，是資料庫初始化用的測試資料。
- 但只要**有人看了某一檔股票**，後端就會去 TWSE 抓那一檔的**真實歷史**，
  把那一檔的測試 K 線**整批換成真實的**（並在 `stock_sync` 表記一筆「已用真實資料」）。
- 所以 seed 的測試 K 線會隨著使用「逐檔被真實資料取代」；
  它的價值是：①一開始就有完整 50 檔可選 ②資料庫離線測試 ③展示帳號。
- `stocks` 的「起始價/波動係數」永遠是測試參數（主要給下單練習參考價與測試函式用），
  跟真實股價無關，這點對展示對象要講清楚。

**對展示對象的標準說法**：
> 「股票清單與產業是真實台股代碼；歷史 K 線在使用者瀏覽該檔時，由後端向
> TWSE 官方抓真實日線存入資料庫並取代初始測試資料；報價是 TWSE 準即時。
> 種子（seed）的測試 K 線只作為資料庫初始化與測試資料；前端 K 線圖目前不再把 seed K 線包裝成真實行情。」

## 3. 首次接手必看的 3 個基本觀念

1. **要同時開兩個程式**：「後端」（背後抓資料、連資料庫）和「前端」（你看到的網頁）。兩個都開著，網頁才有資料。像廚房和外場要同時營業。
2. **「終端機」**：一個可以打指令的黑色文字視窗，用來啟動上面兩個程式。
3. **你會開兩個終端機視窗**：視窗 A 開後端、視窗 B 開前端。**兩個都不能關**，關了程式就停。

## 4. 一次性準備（確認就好，通常已完成）

- ✅ SQL Server 開著（平常用 SSMS 連得進去就代表有開）。
- ✅ 資料表已建（已在 SSMS 跑過 `database/schema.sql`）。
- ✅ 連線設定檔 `backend/.env` 已存在並填好。

> 換新電腦 / 接別人的資料庫，看第 14 節。

## 5. 怎麼打開「終端機」（Windows）

**方法一（最通用）**
1. 按鍵盤左下角 **Windows 鍵**（田字圖案那顆）。
2. 直接打字：`powershell`
3. 按 **Enter**。
4. 跳出一個藍/黑色文字視窗 → 這就是終端機。

**方法二（有用 VS Code 的話）**
VS Code 上方選單 **終端機 / Terminal** → **新增終端機 / New Terminal**。

**建議做法：從專案根目錄開終端機**
1. 用檔案總管打開專案最外層資料夾，也就是看得到 `backend`、`frontend`、`start-all.bat` 的那層。
2. 點一下檔案總管上方的路徑列。
3. 輸入 `powershell`，按 Enter。
4. 這樣打開的終端機會直接位在專案根目錄，後面才能直接使用 `cd backend` 或 `cd frontend`。

> 之後說「開新終端機」就是重複以上動作再開一個。

## 9. 怎麼測試後端有沒有問題（不用裝任何工具）

確定**視窗 A 後端開著**，再開一個新終端機：

```
cd backend
```
```
node test-api.mjs
```

它會自動測 21 項（註冊/登入/問卷/設資金/買/賣/重置/金額對帳）。
**最後出現 `0 失敗 🎉 全部通過` 就代表後端完全正常。**

想在 SSMS 親眼看資料有進資料庫，對 `ncu_db` 執行：
```sql
SELECT COUNT(*) AS 股票數 FROM dbo.stocks;                 -- 應為 50
SELECT COUNT(*) AS K線數 FROM dbo.stock_daily_bars;        -- 應為 15000
SELECT TOP 5 * FROM dbo.orders ORDER BY id DESC;           -- 買賣後會多筆
```

## 11. 出問題對照表

| 你看到的狀況 | 原因 | 怎麼解決 |
|---|---|---|
| 首頁那塊顯示「失敗」紅色 | 後端沒開或視窗 A 被關 | 回視窗 A 重跑 `npm run dev`，網頁按 F5 |
| 網頁打不開 / 一直轉 | 前端視窗 B 沒開好 | 看視窗 B 有沒有網址，沒有就重跑 `npm run dev` |
| `npm 不是內部或外部命令` | 沒裝 Node.js | 到 nodejs.org 裝 LTS 版，重開終端機 |
| `EADDRINUSE` 埠被占用 | 已有同程式在跑 | 關掉舊終端機，或重開電腦再來 |
| 後端 `db ... 連不上` | SQL Server 沒開 / 帳密錯 | 開 SQL Server；檢查 `.env` 的 `DB_*` |
| `Cannot open database ncu_db` | 沒先建表 | 在 SSMS 跑 `database/schema.sql` |
| 買賣金額對帳 FAIL | 沒灌種子 | 視窗 A 先 `node seed.js` 再 `npm run dev` |
| 登入後一直跳問卷 | 正常，第一次要填 | 填完 3 題按完成 |

## 13. 團隊協作注意事項

1. **`backend/.env` 不會上傳 git**（已在 `.gitignore`）。每位開發者 `git pull` 後要**自行建立一份**：複製 `backend/.env.example` 成 `backend/.env`，填入資料庫帳密。
2. **絕不要**把真實密碼貼進任何 `.js` / `.md` / `.example`。
3. 第一次順序：SSMS 跑 `database/schema.sql` → `node seed.js` → `npm run dev`。
4. **K 線圖優先顯示 TWSE 近期真實資料**，完整歷史資料會在背景補齊；若 TWSE 或後端暫時不可用，畫面會誠實顯示資料無法取得。
5. Node 用 v18、`mssql` 鎖 v10，要升級 Node 前先完成團隊版本同步，否則不同開發環境可能跑不動。
6. 資料庫有測試殘留帳號（`test_...@ncu.edu`）不影響展示流程；`node seed.js --force` 只重灌行情三張表、不動帳號。
7. 展示帳號：`demo@invest.ai` / `demo123`。

## 14. 後端檔案清單與連線設定（技術細節）

**檔案角色**

| 檔案 | 做什麼 |
|------|--------|
| `database/schema.sql` | 13 張表建表腳本（在 SSMS 執行） |
| `backend/.env` | 真實連線設定（**不上傳**，每人自建） |
| `backend/.env.example` | 設定範例（可上傳，無真實密碼） |
| `backend/loadEnv.js` | 把 `.env` 載入環境變數 |
| `backend/db.js` | 資料庫連線池、查詢、交易；開機自動補建 `stock_sync` 表 |
| `backend/auth.js` | 密碼加密、登入 token |
| `backend/twseExtra.js` | 抓 TWSE 官方真實歷史日線（STOCK_DAY）＋準即時報價（MIS） |
| `backend/dao.js` | 所有資料庫操作 SQL（含買賣交易、歷史 lazy 入庫、報價快取） |
| `backend/seed.js` | 灌股票/產業/初始測試K線/展示帳號（見第 2-1 節） |
| `backend/server.js` | 後端路由（網址對應功能） |
| `backend/test-api.mjs` | 一鍵測試腳本 |
| `(資料庫) stock_sync` | 記錄每檔歷史「上次同步時間/來源」，由後端開機自動建立，免手動 |
| 前端相關檔案 | 詳見《**前端工程師後端串接說明.md**》（backend API、twseApi.js、前端接法） |

**換電腦 / 接別人的資料庫**：只改 `backend/.env` 裡這段，**程式完全不用動**：
```env
DB_HOST=127.0.0.1     # 改成對方主機 IP
DB_PORT=1433          # 對方 SQL Server 埠號
DB_USER=skyfire       # 對方帳號
DB_PASSWORD=1487      # 對方密碼
DB_NAME=ncu_db        # 對方資料庫名
DB_ENCRYPT=false      # 本機 false；連雲端才 true
DB_TRUST_CERT=true    # 本機自簽憑證 true
AUTH_SECRET=...        # 登入簽章金鑰，正式上線換長隨機字串
```
改完存檔，回視窗 A 按 Ctrl + C 再 `npm run dev` 重啟即生效。

**API 一覽**：
- 需登入（帶 token）：register（註冊）、login（登入）、investor-profile（問卷讀/存）、portfolio（查組合）、portfolio/setup（設資金）、portfolio/reset（重置）、orders/buy（買，含交易）、orders/sell（賣，含交易）
- 公開（不需登入）：health（健康檢查）、market/db-bars/:code（某檔 K 線歷史，第一次會去 TWSE 抓真實歷史入庫；加 `?refresh=1` 強制重抓）、quote/:code（某檔準即時報價，後端 20 秒快取）、twse/market-index（TWSE 最新指數，給驗證面板用）

> API 設計細節另見 `backend/README.md`（給後端工程師或需要檢查 API 設計的人員閱讀，選讀，不影響操作）。
